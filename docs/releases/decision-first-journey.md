# Release Notes — Decision-First Atlas Journey

**Release:** PR35  
**Date:** 2026-04-24  
**Status:** Production default  
**Scope:** Documentation, route verification, and QA only. No feature or behaviour changes.

---

## 1. What Changed

This release formalises the decision-first Atlas journey as the production default.  
The journey surfaces listed below were introduced across PRs 28–34 and are verified ready for live use.

### 1.1 Decision Spine

The engine pipeline now produces a single, ranked decision object before anything is shown to the customer:

```
SurveyDraftInput
  → sanitiseModelForEngine()       (FullSurveyModelV1)
  → runEngine()                    (FullEngineResult)
  → buildScenariosFromEngineOutput()
  → buildDecisionFromScenarios()   (DecisionResult)
  → buildVisualBlocks()            (VisualBlock[])
```

`buildDecisionFromScenarios` selects `recommendedScenarioId`, lifecycle cost, and a `quoteScope` before any presentation layer runs.  
`buildVisualBlocks` converts the decision into ordered, typed blocks (`decision`, `lifecycle`, `quote-scope`, `spatial-proof`, `shower-note`) that every downstream surface consumes without re-running the engine.

### 1.2 Visual Deck (`CanonicalPresentationPage` / `PresentationDeck`)

- Route: `?presentation=1` (full scroll) or `?deck=1` (swipeable)
- Swipeable presentation deck renders all canonical quadrants (House, Objectives, System Performance, Recommendation).
- Every slide declares `canonicalSource.fields` — traceable in DEV via `DevProvenanceBadge` (🔬).
- No `Math.random()` — all outputs are deterministic.

### 1.3 Portal Proof Surface (`CustomerPortalPage`)

- Route: `/portal/:reference?token=<HMAC>`
- Five tabs: Overview, Why, How it works, Future options, Questions.
- Shower compatibility note and spatial proof block are included in the "Why" tab.
- Future tab excludes items already in the quote scope.
- Portal token is generated by `generatePortalToken(reportId)` and validated on arrival.

> **Known limitation:** Portal token/QR generation is a placeholder implementation. Production deployment requires a live HMAC secret in `.dev.vars` / Cloudflare Pages secret store.

### 1.4 Daily-Use Simulator (`ExplainersHubPage` + `SimulatorDashboard`)

- Route: `?lab=1` (direct) or via Survey → Visit Hub → "Open simulator"
- Full interactive proof surface: live taps, heating behaviour, system response.
- Backed by `LifestyleSimulationModule` hourly data — no `Math.random()`.
- Twin Visualiser renders two synchronised 24-hour charts:
  - **Graph 1** (`demandChartData`): area series for Heat (kW) and DHW (kW).
  - **Graph 2** (`chartData`): system response — boiler stepped curve / HP horizon / hot-water reserve.
- Cooling uses `T(t) = T_outdoor + (T_initial − T_outdoor) × e^(−t/τ)`.
- Heating uses stepped 30 kW sprint logic.

### 1.5 Engineer Handoff (`EngineerPreinstallPage`)

- Route: `/visit/:visitId/engineer`
- Shows recommended scenario, quote scope, shower compatibility note, spatial layout, and discharge routes.
- Driven by `buildEngineerHandoff()` and `buildEngineerLayout()` — same engine output, no re-derivation.
- `buildLayoutSummaryLines()` renders a dense, print-first summary for the installer.

### 1.6 Planner Readiness / Guided Survey (`FloorPlanBuilder` + `GuidedSurveyChecklist`)

- Route: Floor Plan tool within Visit journey, or `?floor-plan-tool` path
- `validatePlanReadiness()` checks rooms, boiler object, flue, cylinder, and discharge route before handoff.
- `GuidedSurveyChecklist` steps through object placement with room-type heuristics.
- `guidedSurveySteps.ts` drives the ordered checklist — no freeform CAD.

> **Known limitation:** The planner is not a full CAD environment. Object sizes are defaults requiring site verification. LiDAR/native scan integration is handled separately via the Atlas Scan import boundary (`?scan-package=1`).

### 1.7 Advice Pack Print Output (`CustomerAdvicePrintPack`)

- Route: `?print=survey` (default customer output)
- Rendered from `VisualBlock[]` truth — decision, lifecycle, quote scope, portal CTA.
- No diagnostics or internal fields are visible to the customer.
- Portal CTA includes a signed URL + QR code pointing to `/portal/:reference`.
- Internal diagnostic version (`CustomerRecommendationPrint`) is only reachable via `?print=survey&internal=1` — never the default.

## 3. Known Limitations

The following items are explicitly out of scope for this release. They are documented here to avoid ambiguity.

| Limitation | Detail | Planned resolution |
|---|---|---|
| Planner is not full CAD | `FloorPlanBuilder` provides a guided room-and-object survey, not architectural drawing. Rooms are placed with tile/grid snapping; no free-form walls. | Separate LiDAR / scan integration milestone. |
| LiDAR / native scan integration is separate | Atlas Scan (`?scan-package=1`) can import a scan bundle, but the Spatial Twin does not yet auto-populate the planner from point-cloud data. | Atlas Scan import roadmap. |
| Portal token / QR generation is a placeholder | `generatePortalToken()` uses a placeholder HMAC. A production HMAC secret must be set in `.dev.vars` (local) and Cloudflare Pages secrets (production) before QR codes are live. | Infrastructure ticket. |
| Default object sizes require site verification | `FloorPlanBuilder` assigns default dimensions for boiler, cylinder, and flue objects. These defaults are typical UK values and must be confirmed against the installed appliance datasheets on site. | Surveyor sign-off process. |
| Assumed routes need verification | Discharge routes in the planner default to `assumed: true` until a surveyor confirms the physical route. Engineer handoff labels assumed routes accordingly. | Site survey confirmation. |

## 4. Manual QA Script

Run these steps end-to-end on a staging or local build before any production deployment.

### Prerequisites

```
npm run build
npm run preview   # or: wrangler pages dev dist
```

Navigate to `http://localhost:8788` (or the staging URL).

---

### Step 1 — Create or load a survey

1. Open the application root (`/`).
2. Click **Start new visit** → enter an optional reference number → confirm.
3. Complete the full survey (all steps, including Heat Loss and Priorities).
4. On the Quotes step, add at least one contractor quote.
5. Confirm the visit is saved (Visit Hub shows a "Complete" badge or working payload).

**Acceptance:** Visit Hub shows the visit with a working payload and navigation options.

---

### Step 2 — Run recommendation

1. From the Visit Hub, click **View recommendation** (or **Open presentation**).
2. Verify the `CanonicalPresentationPage` loads with four quadrants: Your House, Your Objectives, System Performance, Recommendation.
3. In DEV builds, confirm 🔬 provenance badges appear on each slide.
4. Check the recommended system label matches the engine output for the entered survey data.

**Acceptance:** Deck renders, all four quadrants populated, no blank slides.

---

### Step 3 — Open advice pack

1. From the Visit Hub, click **Print summary**.
2. Wait for the `?print=survey` route to render (`CustomerAdvicePrintPack`).
3. Verify:
   - Decision block is present with a clear recommendation.
   - Lifecycle cost block is present.
   - Quote scope block is present and shows included items.
   - Portal CTA appears at the bottom with a QR code / link.
   - No diagnostic or internal fields are visible.
4. Confirm `?print=survey&internal=1` renders `CustomerRecommendationPrint` (old internal format, not the default).

**Acceptance:** Advice pack renders correctly; internal route is guarded.

---

### Step 4 — Open portal

1. From the advice pack or Visit Hub, follow the portal link.
2. Confirm URL pattern is `/portal/:reference?token=<HMAC>`.
3. Verify five tabs render: Overview, Why, How it works, Future options, Questions.
4. On the **Why** tab, confirm shower compatibility note and spatial proof block are visible (if applicable).
5. On the **Future** tab, confirm items already in the quote scope are excluded.

**Acceptance:** All five portal tabs render; content matches the engine recommendation.

---

### Step 5 — Run simulator

1. From the Visit Hub, click **Open simulator** (or navigate via `?lab=1`).
2. Confirm `ExplainersHubPage` / `SimulatorDashboard` loads.
3. Toggle a tap on and off — verify system response chart updates.
4. Verify Twin Visualiser shows two synchronised 24-hour charts (demand and system response).
5. Confirm no `Math.random()` jitter in the charts (re-opening the same data should produce identical curves).

**Acceptance:** Simulator loads, interactive controls respond, charts are deterministic.

---

### Step 6 — Open engineer handoff

1. From the Visit Hub, click **Engineer route** (or navigate to `/visit/:visitId/engineer`).
2. Verify the page shows:
   - Recommended system (matches Step 2).
   - Quote scope (matches Step 3).
   - Shower compatibility note (if applicable).
   - Spatial layout summary (rooms, boiler, cylinder, flue).
   - Discharge route (labelled "assumed" if not confirmed).

**Acceptance:** Engineer handoff page renders with consistent data.

---

### Step 7 — Open planner preview

1. During or after the visit survey, access the Floor Plan tool.
2. Place at least one room, one boiler object, one cylinder, and a discharge route.
3. Confirm `validatePlanReadiness()` status updates (guided checklist turns green as items are placed).
4. Verify default object sizes are shown (note: require site verification).

**Acceptance:** Planner tool loads; guided checklist tracks object placement; readiness indicator updates.

---

### Step 8 — Verify route and object readiness

1. Open `/visit/:visitId/engineer` — confirm assumed routes are labelled.
2. Open `?visit-handoff=1` — confirm `VisitHandoffReviewPage` loads the sample fixture.
3. Open `?customer-share=1` — confirm `CustomerSummaryPrintPage` renders.
4. Open `?engineer-share=1` — confirm `EngineerSummaryPrintPage` renders.
5. Open `?internal=1` on the root — confirm `DevMenuPage` does **not** appear (no `?devmenu=1` flag).
6. Open `?print=survey&internal=1` — confirm old diagnostic format is visible but not the default.

**Acceptance:** All surface routes reach the correct component; guarded routes require the correct flag.

## 6. Terminology Reminder

All user-facing copy in this release uses only terms from `docs/atlas-terminology.md`. In particular:

- ✅ **On-demand hot water** — water heated only when a tap is opened (combi boiler).
- ✅ **Stored hot water** — hot water drawn from a pre-heated cylinder.
- ✅ **Mains-fed supply** — supply at mains pressure (not "high pressure system").
- ✅ **Tank-fed hot water** — supply from a cold water storage tank (not "gravity system").
- ✅ **Simulator / System Simulator / System Lab** — only for `ExplainersHubPage` + `SimulatorDashboard`.
- ✅ **Scenario Preview / Behaviour Preview** — for lightweight read-only behaviour card panels.
